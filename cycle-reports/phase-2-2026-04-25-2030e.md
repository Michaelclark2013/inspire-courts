## Note
Despite the "Functionality" phase label, the work this iteration was the
email-verification token-at-rest hardening identified during last cycle's
phase-6 audit. Logically a security fix, but it lives at the auth-flow
correctness layer and pairs naturally with the existing email-verification
flow rather than waiting until phase 6.

## Gap fixed
`users.emailVerifyToken` stored raw 32-byte hex tokens, with the same
profile as the just-fixed `reset_tokens` flow: a users-table read leak
would let an attacker mark arbitrary accounts as verified or claim
pending-verification accounts.

Added `hashVerifyToken(token)` helper to `src/lib/email-verification.ts`
(SHA-256). Updated:
- `register` flow — stores `hashVerifyToken(verifyToken)`, emails the raw token
- `resend-verification` flow — same write path
- `verify-email` lookup — hashes incoming token before DB lookup

Existing in-flight registrations become unable to verify with their
old emailed link (they'd need to "Resend"), but the 24-hour TTL
limits exposure.
