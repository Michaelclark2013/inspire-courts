## Current state
The forgot-password / reset-password flow stored raw 32-byte hex reset
tokens directly in the `reset_tokens` table. If an attacker ever gained
read access to the DB (any of: SQL injection, backup leak, internal
misconfig, restored prod DB into a non-prod environment), they could
walk the table and use any non-expired raw token to reset any user's
password. The standard mitigation is to store the SHA-256 hash of the
token in the DB and only ever email the raw token to the user — same
shape (64-char hex), no schema change.

Also affects `/api/admin/users/[id]/reset-password` which uses the same
table via the same shape.
