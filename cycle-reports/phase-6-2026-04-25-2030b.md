**Phase scope:** OWASP top 10, secrets, dependencies, CSP, auth hardening, rate limiting, PII.

## Current state
Audited public API surface for missing rate limits, missing auth, and IDOR
vectors. All routes have rate limits or auth. Found one IDOR:

`/api/tournaments/[id]/registration-status?reg=N` accepted both a path
tournament id and a query reg id but never verified they paired. The query
was `WHERE id = regIdNum` only — the tournament path segment was consumed
and discarded. An attacker could iterate `?reg=1,2,3,...` and read out
team name + status + paymentStatus for every registration in the system,
across every tournament.

## Gaps found this iteration

### P0 — fix this iteration
1. **IDOR in registration-status endpoint** —
   `src/app/api/tournaments/[id]/registration-status/route.ts`. Add
   `tournamentId` to the WHERE clause so callers must know *both* the
   tournament id and the reg id, and the pair must match. The only known
   caller (the post-registration confirmation page) already passes both,
   so this is backward-compatible.
